# ADR-0004: ed25519 signing now, cosign/Sigstore-compatible later

- Status: Accepted
- Date: 2026-06-10

## Context

The lockfile and individual artifact approvals should be signable so a team can
attest "we reviewed exactly these bytes." The mature ecosystem answer is
cosign/Sigstore keyless (OIDC) signing, but it adds significant dependencies and
infrastructure that would block the MVP.

## Decision

For the MVP, sign with **detached ed25519 signatures** over canonical bytes
(the artifact content hash, or the lockfile body), encoded as
`ed25519:<base64>`. The scheme is self-describing via its prefix.

The capability lives behind the `ports.Signer` interface so a future
cosign/Sigstore adapter can satisfy the same port with a different prefix,
without changing callers.

## Consequences

- Signing/verification ships with `crypto/ed25519`, no external dependencies.
- The signature format leaves room for additional schemes (`cosign:…`).
- Key management (storage, a trust registry, rotation) is intentionally minimal
  in the MVP and expands with the control plane (Component 3).
